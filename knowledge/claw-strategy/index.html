CHAPTER 01

发生了什么？

春节前后，公司内出现了一波"养虾"热潮。多个团队各自搭建了自己的 Claw（小龙虾）平台，用户数和 Token 消耗都在快速增长。
用手机的比喻来说：4 个部门各自造了一台功能机——每台都能打电话，但品牌不同、充电线不通、通讯录不互通。

📱 贯穿全文的类比

想象一下：公司里有 4 个部门，各自造了一台功能机。每台都能打电话发短信，但系统不通、数据不通、充电线都不一样。
员工手里攥着好几台手机，不知道用哪个。IT 部门也管不住安全和费用。

而正确的做法是：由一个部门造一台 iPhone——统一的硬件、统一的操作系统、统一的安全体系。
然后所有人在上面造 App：公司提供预装应用，各业务线开发专属工具，每个人也能装自己的 App。

接下来的每一章，都可以用这个类比来理解。

CHAPTER 02

分散建设，问题在哪？

分散建设不是"百花齐放"，而是"各自为战"。它正在带来三个层面的风险。
📱 用手机类比：这就是每台功能机各自带来的问题。

🔒 风险一：安全失控

各团队自行把控数据安全，没有统一标准。一旦某个产品出现数据泄露，影响的是整个公司的信誉。

"将业务数据用 AI 做处理分析，符合数据安全规定么？这个信息规范从哪里来，现在靠口口相传。"—— 员工反馈

各产品的安全标准不一、审计能力不同、Skill 审核机制缺失或宽松。敏感数据的边界全凭各团队"自觉"，无法统一追溯。

🔑 风险二：权限割裂

AI 无法继承用户的已有权限，内部系统对接各做各的，导致重复建设且体验差。

"内部工具多算是一个困扰，希望集中力量各个平台都打通，现在是四处圈地的逻辑。"—— 员工反馈

AI 读不了 C4 文档、看不了数据看板、找不到准确的数据表口径。各产品各自对接内部系统，进度和质量参差不齐。

💰 风险三：费用失控

多个产品各自消耗 Token，无法统一核算 ROI，也无法通过模型路由来优化成本。

4 个产品单日 Token 成本约 2.5 万元且快速增长。每个产品固定使用 1-2 个模型，无法根据任务复杂度选择性价比更高的模型。

❌ 继续分散建设的后果
  • 安全事件无法追溯（哪个 Claw 泄露了数据？）
  • 用户在多个工具间反复切换
  • 优秀 Skill 无法跨产品复用
  • Token 成本持续膨胀，无人负责
  • 各团队疲于维护基础设施，无力创新
✅ 统一建设的收益
  • 统一安全基线，Skill 审核一道关
  • 一个入口，降低用户学习成本
  • Skill 市场共享，避免重复造轮子
  • 智能模型路由，在保效果的前提下降本
  • 团队专注场景创新，不再重复建基础设施
📱 手机类比

功能机的三大痛点：
安全：有的功能机连锁屏密码都没有，丢了手机就丢了所有数据。
权限：每台手机各存一份通讯录，换手机要重新导入，数据永远不同步。
费用：每台手机各自买话费套餐，公司无法统一管理和优化。

iPhone 怎么解决？ 统一的安全体系（Face ID + 远程锁定）、统一的账号（iCloud 同步所有数据）、统一的采购（企业批量管理）。
同理，只有统一建设才能从根本上解决这三个问题。

CHAPTER 03

为什么应该由研发线主导？

研发线同时拥有"底层基础设施"和"最强 Agent 能力"两张牌，是唯一能从下到上贯通的团队。
📱 谁能同时造芯片和操作系统，谁就该造这台 iPhone。

⚙️ 优势一：万擎平台（MAAS）—— iPhone 的芯片

万擎平台是公司的模型服务基础设施，提供：

  • 统一模型调用：接入万擎、Claude 等多种模型
  • 算力资源：推理节点、GPU 调度
  • 安全保障：租户级数据隔离、访问控制、审计日志
  • 成本优化：按任务智能路由到最合适的模型
🧠 优势二：Code Agent —— iPhone 的 iOS

Code Agent 是能力天花板最高的 Agent 类型。它不仅能执行已有的任务，还能通过编程创造新的能力。
📱 这就是功能机和 iPhone 的核心区别——功能机只能用出厂预装的功能，iPhone 能安装无限的 App。

📱 手机类比

功能机能做的事，出厂就定了——电话、短信、贪吃蛇。想加个微信？不好意思，做不到。
iPhone 的革命性在于：缺什么功能，装个 App 就行。
Code Agent 也一样：缺什么能力，写段代码就有了。这就是为什么它的能力是无限的。
普通 Agent = 功能机（能力有限集合），Code Agent = 智能手机（能力无限集合）。

📊 优势三：模型管理 —— 智能"派单"降成本

研发线管理模型，可以根据任务复杂度智能路由到不同成本的模型：

任务类型匹配模型成本示例
复杂推理Claude / 万擎大杯$$$架构设计、复杂代码
常规分析万擎中杯$$SQL 生成、文档摘要
简单对话万擎小杯$日常问答、格式转换
📱 手机类比

就像 iPhone 的 App Store 智能推荐：轻量任务推荐免费 App，专业需求推荐付费 App。
AI 任务也一样——日常问答用轻量模型（便宜），复杂推理才用旗舰模型（贵但效果好）。
分散建设做不到这种统一调度，就像每台功能机都只能绑定一个运营商。

🏪 优势四：三层 Skill 与知识生态

研发线牵头建设公司级 → 业务级 → 团队/个人级的 Skill 和知识中心，形成完整的 AI 能力生态。

1

公司级 Skill & 知识（默认生效）

研发线有 Agent 评测能力，持续筛选效果最好的技能和知识，让所有人默认就能用上最优方案，无需个人配置

2

业务级 Skill & 知识（合作共建）

研发线与各业务线合作，共建业务场景专属的 Skill——比如电商场景的商品分析、客服场景的话术助手等

3

团队/个人级 Skill & 知识（开放生态）

开放平台能力，每个团队甚至每个人都可以建设自己的 Skill，满足个性化需求

📱 手机类比

这就是 iPhone 的 App Store 模式。
公司级 Skill = 手机预装应用（电话、浏览器、地图，人人都有）
业务级 Skill = 行业定制应用（企业微信、专业工具，按需安装）
个人级 Skill = 个人下载的 App（游戏、健身，自由选择）
研发线建的就是这部 iPhone + App Store，而其他团队只是各自做了一台功能机。

CHAPTER 04

具体怎么做？

统一建设有多种路径。下面先对比三种可选方案，再展开推荐方案的具体架构。
📱 继续用手机来理解。

三种建设方案对比
对比维度 方案 A：维持现状 方案 B：选一个现有产品统一 方案 C：研发线新建统一平台
做法 各团队继续各做各的 选 MarioClaw 或 LangBridge 等某个现有产品做公司标准 研发线以 Code Agent + 万擎平台为基础，建设统一 Claw 平台
安全保障 各自把控，标准不一 取决于选中产品 万擎平台统一保障
成本优化 无法统一优化 有限优化 智能路由 + 集中管控
Agent 能力 各产品能力参差 受限于选中产品的能力天花板 Code Agent，能力无上限
扩展性 各自扩展，无法复用 依赖 Skill 人工开发 编程扩展，缺什么写代码补
落地难度 无需改变 需迁移 + 协调 需建设 + 迁移
长期风险 安全/成本不可控 能力受限于选中产品 体系最完整，风险最低
📱 手机类比

方案 A（维持现状）= 继续用 4 台不同的功能机，互不兼容。
方案 B（选一个产品统一）= 选最好的那台功能机发给所有人——但诺基亚 N95 做到极致，也打不过 iPhone 1 代。功能机就是功能机，装不了 App。
方案 C（研发线统一建设）= 造一台 iPhone——统一的操作系统、开放的 App 生态、可持续进化。
短期方案 C 投入大一点，但它是唯一能持续进化的方案。

为什么方案 B（选一个现有产品统一）不够好？

表面上看，方案 B 更快——直接选一个已有产品推广全公司。但存在三个核心问题：

  • 能力天花板低：现有产品都是对话型 Agent，能力受限于已有 Skill，无法通过编程扩展
  • 底层不可控：没有万擎平台做底座，安全、成本优化、模型路由都需要额外建设
  • 扩展成本高：每新增一个场景都需要人工开发 Skill，而 Code Agent 可自主编程解决

一句话：方案 B 解决了"统一"问题，但没有解决"能力"问题。方案 C 既统一又最强。

⭐ 推荐方案：研发线统一建设（方案 C）

以 Code Agent 为核心，万擎平台为底座，构建三层架构。

统一架构：三层能力体系
1

入口层：云端 + 端侧

Kim 侧边栏覆盖日常办公 · KIM 机器人覆盖IM场景 · 本地客户端覆盖深度任务（用户按习惯使用即可）

2

核心层：Code Agent 运行时

代码生成/理解/修复 · 统一 Skill 市场 · 缺什么能力就写代码补

3

基础层：万擎平台（MAAS）

模型调用 · 智能路由 · 成本优化 · 数据隔离 · 审计日志

📱 手机类比

这三层架构就是一部 iPhone：
入口层 = 手机屏幕（用户看到和操作的界面，多种交互方式）
核心层 = iOS 操作系统（决定手机能装什么 App、做什么事）
基础层 = 自研芯片 A 系列（性能、安全、效率的根基）
研发线 = Apple：同时拥有芯片和操作系统，所以只有它能造出 iPhone。

三大隐患的解决方案
🔒 安全隐患
  • 各团队安全标准不一
  • Skill 审核机制缺失
  • 数据泄露难以追溯
  • "规范靠口口相传"
✅ 研发线方案
  • 万擎平台提供租户级数据隔离
  • 统一 Skill 审核，上架前安全检查
  • 统一日志 + 异常检测 + 事件追溯
  • 输出公司级 Agent 安全规范
🔑 权限隐患
  • AI 无法继承用户权限
  • 内部系统各自对接
  • 数据表口径不统一
  • "四处圈地"体验割裂
✅ 研发线方案
  • 对接统一身份认证，继承用户权限
  • 统一 API 对接层，一次建设全局复用
  • 建设企业知识库，标准化数据定义
  • 一个入口统一体验
💰 费用隐患
  • Token 消耗分散，无法统一核算
  • 各团队固定模型，无法优化
  • 分散采购，议价能力弱
  • 成本快速增长，无人负责
✅ 研发线方案
  • 统一核算，按团队/场景清晰归因
  • 智能路由，按任务复杂度选模型
  • 配额管理 + 超额预警
  • 集中采购，规模议价
📱 核心逻辑

安全、权限、费用这三个问题，本质上都是"需要统一管控"的问题。
功能机永远解决不了这些——就像你不可能给 4 台不同品牌的功能机统一装一个安全系统。
只有造一台 iPhone，把安全、账号、费用都内置在操作系统层，才能从根本上解决。

CHAPTER 05

结论与建议

四个核心决策，一句话总结。

📌 核心结论

应该统一建设，由研发线主导。
依托万擎平台保障安全和成本，以 Code Agent 为核心提供最强能力。
这是能力最强、成本最优、安全最可控的方案。

📱 一句话：不要让 4 个部门各造功能机了，由研发线造一台 iPhone，然后所有人一起造 App。
